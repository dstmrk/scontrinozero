### Batch 1 — Export CSV (core pipeline)

File esaminati: `src/lib/csv.ts`, `src/lib/csv.test.ts`, `src/lib/receipts/csv-export.ts`, `src/lib/receipts/csv-export.test.ts`, `src/app/api/export/receipts/route.ts`, `src/app/api/export/receipts/route.test.ts`, `src/app/dashboard/storico/export-csv-button.tsx`, `src/app/dashboard/storico/export-csv-button.test.tsx`, `src/app/dashboard/storico/page.tsx`, `src/types/storico.ts`, e (referenziati) `src/components/storico/storico-client.tsx`, `src/lib/date-utils.ts`, `src/lib/receipts/document-lines.ts`.

#### Findings

1. **major** — `src/lib/receipts/csv-export.ts:90,29` — Mismatch colonna vs filtro. La query forza `kind = "SALE"` (riga 90) ma il CSV include la colonna `id_documento_annullato` (riga 29, popolata solo su righe `kind = "VOID"`). La colonna è quindi **sempre vuota** nell'export, anche per record `VOID_ACCEPTED` (che semanticamente sono SALE annullate). Per dare valore alla colonna serve o (a) rimuoverla, o (b) includere `kind = "VOID"` con kind-aware rendering, o (c) `LEFT JOIN` su `commercial_documents AS void_doc ON void_doc.voided_document_id = commercial_documents.id` per popolare la colonna sui record `VOID_ACCEPTED`. Aggiungere test che documenti il comportamento atteso. Bloccante per l'UX, non per la correttezza.

2. **major** — `src/lib/receipts/csv-export.ts:128-131` — OFFSET pagination cumulativa O(n²) per export grandi. Su un Pro con 50k scontrini gli ultimi batch fanno ~5000 row-scan ciascuno; il problema è esattamente quello chiamato out in PLAN.md backlog P2 ("Paginazione cursor-based su storico, export e Developer API"). Per i volumi attuali (alpha/beta) non è urgente, ma il pattern keyset è banale da introdurre subito mentre l'API è ancora interna: `where(and(...conds, lt(createdAt, cursor)))` con cursor estratto dall'ultimo row del batch precedente. Decidere se anticipare o segnalare nel CHANGELOG come known-limitation della v1.3.0.

3. **minor** — `src/lib/receipts/csv-export.ts:129` — OFFSET pagination senza tie-break secondario su `id`. Con scritture concorrenti durante un export (un nuovo SALE entra fra un batch e il successivo) lo stesso documento può comparire due volte o essere saltato perché lo shift di `createdAt` ripiega l'offset. Mitigare con `.orderBy(desc(createdAt), desc(id))`. Probabilità reale bassa (export raramente concorrenti con emissioni), ma costo zero.

4. **minor** — `src/app/api/export/receipts/route.ts:67-68` — Non-null assertion `user!.id` con commento giustificativo. Funziona ma è un code smell. Refactor preferibile: estrarre un early return `if (!user) return errorJson(401, "Non autenticato.")` PRIMA di `assertProPlan`, oppure rendere il tipo di ritorno di `assertProPlan` un discriminated union dove `ok: true` implica `user` non-null (rifletterebbe già l'invariante implementato).

5. **minor** — `src/app/api/export/receipts/route.ts:21-33` + `87-95` — Validazione data fatta due volte (regex Zod + `parseStrictIsoDateUtc`). Difesa-in-profondità OK ma il path "regex pass / strict-parse fail" produce un secondo errore 400 con messaggio identico — codice ridondante. Sostituibile con un singolo Zod `.refine(v => parseStrictIsoDateUtc(v) !== null, "...")`. Tenere se l'errore Zod è più chiaro all'utente, altrimenti consolidare.

6. **minor** — `src/app/dashboard/storico/page.tsx:39-43` — Calcolo default "ultimi 7 giorni" usa `today.toISOString().split('T')[0]` (UTC) invece del fuso fiscale Europe/Rome. Inconsistente con la fix recente `ddeefe4` che ha ancorato i bucket analytics a Europe/Rome. Tra mezzanotte UTC e l'1/2 di mattino italiano il default sarebbe "ieri" anziché "oggi". Usare `formatInTimeZone(new Date(), "Europe/Rome", "yyyy-MM-dd")` (date-fns-tz è già nelle deps per la fix analytics). Stessa logica replicata in `storico-client.tsx` per i fallback del DateRangePicker (verificare nel Batch successivo).

7. **nit** — `src/lib/csv.ts:61` — `FORMULA_LEADERS.has(str[0])`: per stringhe che iniziano con surrogate-pair `str[0]` restituisce il primo code-unit (half), non il code-point. Nessun rischio reale (i leader pericolosi sono tutti ASCII) ma per pulizia `str.codePointAt(0)` o un controllo `str.startsWith(c)` sui leader.

8. **nit** — `src/lib/receipts/csv-export.ts:42` — `publicRequest: unknown` perde tipizzazione del payload. Drizzle expone già `InferSelectModel`; un alias come `Pick<SelectCommercialDocument, 'publicRequest'>` o uno Zod schema dedicato chiarirebbe la shape. Compromesso accettabile, l'estrattore `extractPaymentMethod` è già difensivo.

9. **nit** — `src/lib/receipts/csv-export.test.ts:224-228` — `mockCalcDocTotal.mockImplementation` restituisce sempre `10` su entrambi i branch del ternario → il test non distingue gli output per documento. Il test passa ma non sta verificando la differenziazione. Cosmetico.

10. **nit** — `src/app/api/export/receipts/route.ts:39,122` — `buildFilename(from = "tutti", to = "tutti")` con magic string italiana ripetuta nel default. Estrarre in costante `FILENAME_ALL_FALLBACK = "tutti"` per coerenza e futura i18n. Trascurabile.

#### Note positive

- Escaping CSV RFC 4180 corretto, inclusa la protezione formula-injection (`=`, `+`, `-`, `@`, `\t`, `\r`) con apostrofo, e doppia-quote interna raddoppiata.
- BOM UTF-8 emesso prima dell'header (compat Excel italiano).
- Streaming via `ReadableStream` con `controller.error()` su errori DB: il client riceve un download troncato e percepisce il fallimento (no fake-success).

- Tutti i test hanno `expect()`, mock `RateLimiter` usa `function`, variabili `mock*` prefissate, validazione UUID irrilevante qui (boundary GET su query string strettamente validata).
- Pro gate enforcement server-side prima del rate limit: nessun bypass possibile dal client.
- `parseStrictIsoDateUtc` con round-trip check su year/month/day previene la normalizzazione JS di date impossibili (es. 2026-02-31).

### Batch 2 — Analytics backend

File esaminati: `src/server/analytics-actions.ts`, `src/server/analytics-actions.test.ts`, `src/server/analytics-helpers.ts`, `src/app/dashboard/analytics/page.tsx`, e (referenziato) `src/lib/server-auth.ts` (solo per uso di `checkBusinessOwnership`).

#### Findings

1. **major** — `src/server/analytics-actions.ts:44-64` + `133-223` + `src/app/dashboard/analytics/page.tsx:51-55` — Amplificazione massiccia di auth/plan/DB per render: ogni delle 3 server action (`getAnalyticsKpis`, `getRevenueTimeseries`, `getPaymentBreakdown`) invoca `authorizePro` che fa `getAuthenticatedUser` + `checkBusinessOwnership` + `getPlan`, poi `fetchSaleDocsInRange` + `computeTotalsByDoc` (fetch lines). La pagina chiama le tre action in `Promise.all` e a sua volta esegue `getAuthenticatedUser` + `getPlan` per il gate UI. Conteggio totale per una sola visita: **4× `getPlan`** (1 in page + 3 in actions), **3× `checkBusinessOwnership`**, **3× docs query**, **3× lines query**, **4× Supabase auth call**. Per un Pro con range 90d ognuna delle 3 docs query scansiona migliaia di record. Fix consigliato: introdurre `getAnalyticsDataset(businessId, range)` (1 chiamata auth + 1 docs + 1 lines) e derivare i 3 output in memoria; oppure introdurre un `AnalyticsContext` cacheato per request via `react/cache`. Impatta latency percepita (priorità #1 del progetto) e costi DB.

2. **major** — `src/app/dashboard/analytics/page.tsx:57-63` — Silent error masking: `safeKpis = "error" in kpis ? ZERO_KPIS : kpis` (e analoghi su timeseries/breakdown) mostrano zeri senza alcun segnale visivo se una qualunque action torna `{error}`. La gate `getPlan` precedente rende lo scenario raro (downgrade nel mezzo di un render, race con auth refresh), ma se accade è invisibile e produce metriche fasulle viste dall'utente come reali. Minimo: `logger.warn` con `userId` e `businessId` e fallback UI esplicito ("Impossibile caricare i dati, riprova"). Decidere se davvero `ZERO_KPIS` è il fallback corretto.

3. **major** — `src/server/analytics-actions.ts` (tutto il file) — Nessun rate limit sulle server action analytics. Un Pro autenticato può spammare le 3 query (range 90d può scansionare migliaia di documenti) e degradare il DB. Le altre operazioni costose seguono già la convention CLAUDE.md (`emit:<userId>` 30/h, `void:<userId>` 10/h, `csv:<userId>` 10/h, `pdf:<ip>` 60/h). Aggiungere `analytics:<userId>` (suggerito 60/h come `pdf:`, oppure 120/h se le visite tipiche fanno polling). Istanziare il `RateLimiter` a livello modulo e applicarlo PRIMA di `fetchSaleDocsInRange`. Per-userId, non per-businessId, per ridurre l'attack surface.

4. **minor** — `src/server/analytics-actions.ts:66-73` — `validateRange` è dichiarata `async` ma non contiene `await`. Pattern coerente con `authorizePro` ma costa un microtask gratis a chiamata. Render sync: `function validateRange(range): { ok: true } | { ok: false; error: string }`.

5. **minor** — `src/server/analytics-actions.ts:101-112` — Query docs unbounded sul range: per 90d × N scontrini/giorno il risultato non ha `LIMIT`. PLAN.md backlog P2 cita esattamente `searchReceipts`/`exportUserData` ma non `analytics-actions` (la feature è nuova in v1.3.0): andrebbe aggiunto al backlog o mitigato con un safety `LIMIT 50_000`. In più: nessun `withStatementTimeout` wrap (altri server action usano `db-timeout.ts` per gli stessi pattern di scansione lunga). Aggiungerlo per non lasciare connessioni pinned su query degenere.

6. **minor** — `src/server/analytics-actions.ts:45-50` — `try { await getAuthenticatedUser() } catch { return { error: "Non autenticato." } }`. La catch-all converte qualsiasi errore (network blip Supabase, errore di parsing del JWT) in "Non autenticato" — fuorviante per il debugging e per il logging. Filtrare per il tipo d'errore atteso (e.g. `AuthRequiredError`) e rilanciare gli altri; oppure `logger.warn({err, businessId}, "authorizePro: unexpected error")` prima del return. Coerente con CLAUDE.md regola 23.

7. **minor** — `src/app/dashboard/analytics/page.tsx:23-55` — `getPlan` chiamato in page (riga 28) E poi 3× dentro le action via `authorizePro` = 4 chiamate per request. Mitigation: cache via `react/cache` su `getPlan` (vale anche per altre pagine che lo chiamano), oppure passare il `plan` già risolto alle action come parametro. Connesso al finding #1 di questo batch.

8. **minor** — `src/app/dashboard/analytics/page.tsx:51-55` — Range iniziale hardcoded a `"30d"`, `searchParams` non letti. Conseguenza: `/dashboard/analytics?range=90d` non è un deep link e al refresh il client perde lo stato del range scelto. Incoerente con `dashboard/storico/page.tsx` che legge `dal`/`al`/`stato` dai searchParams (vedi Batch 1). Allineare i pattern.

9. **nit** — `src/server/analytics-actions.ts:113` — `rows as DocRow[]` per la conditional spread di `selection` (la versione "con publicRequest" e quella "senza"). Eliminabile con un overload o due funzioni separate; cast è pragmatico ma sopprime un eventuale errore di shape.

10. **nit** — `src/server/analytics-helpers.ts:92-100` — Defensive `parts.find(...)?.value ?? "0"`: se Intl restituisse un set parti inatteso il fallback "0" produrrebbe noon UTC = giorno fiscale sbagliato. È un caso teorico, ma un `throw` esplicito sarebbe più safe (Intl non dovrebbe mai variare, e se variasse vogliamo saperlo).

11. **nit** — `src/server/analytics-helpers.ts:157-163` — `fillMissingDays` confronta `cursor < end` (string lex). Funziona perché `yyyy-MM-dd` è lex-sortable, ma non c'è un commento che documenti l'invariante (qualcuno potrebbe cambiare il formato a `dd-MM-yyyy` localizzato e rompere silenziosamente la loop terminale).

12. **nit** — `src/app/dashboard/analytics/page.tsx:44` — `<div />` come child obbligato di `ProFeatureGate`. Vuoto e senza semantica. Decidere: rendere `children` opzionale nel componente gate, oppure usare un `<></>` Fragment esplicito. Probabilmente da rifinire nel Batch 4 (revisione di ProFeatureGate).

### Batch 3 — Analytics frontend (UI + recharts)

File esaminati: `src/components/analytics/analytics-client.tsx`, `src/components/analytics/kpi-cards.tsx`, `src/components/analytics/kpi-cards.test.tsx`, `src/components/analytics/payment-breakdown.tsx`, `src/components/analytics/revenue-sparkline.tsx`, e (referenziato) `src/lib/utils.ts` (`formatCurrency`/`formatDate`).

#### Findings

1. **major** — `src/components/analytics/analytics-client.tsx:56-70` — Race condition su cambio range rapido. `handleRangeChange` lancia 3 server action in `Promise.all` dentro `startTransition`, senza cancellation né tracking della richiesta più recente. Se l'utente cambia range due volte velocemente (7d → 90d), il risultato della prima richiesta (più lenta) può arrivare dopo la seconda e sovrascrivere `kpis`/`timeseries`/`breakdown` con i dati sbagliati. React `useTransition` segnala solo `isPending`, non cancella. Soluzioni: (a) `useRef<AnalyticsRange>` per latest, ignorare i resolve se `nextRange !== latestRangeRef.current`; (b) `AbortController` (richiede che le server action accettino signal — non oggi); (c) **TanStack Query** (già nelle deps) con `queryKey: ['analytics', businessId, range]` gestisce nativamente latest-only + cache fra switch (7d → 30d → 7d torna istantaneo). Opzione (c) preferibile, coerente con tech stack documentato.

2. **major** — `src/components/analytics/revenue-sparkline.tsx:33,49-53` — Timezone bug client-side. `formatDate(value)` fa `new Date("2026-05-19").toLocaleDateString("it-IT")` SENZA `timeZone`. Il server bucket-ifica per giorno fiscale Europe/Rome (fix `ddeefe4`), ma `toLocaleDateString` usa la tz del browser. Per un browser configurato in tz != Europe/Rome (es. mobile in roaming oltre US) il `2026-05-19` (UTC midnight) viene re-interpretato come `2026-05-18T20:00-04:00` e mostrato come "18/05/2026". Inconsistente con la fix server. Fix: `formatDate(value, "numeric", "Europe/Rome")` (estendendo l'API) oppure trattare la stringa come calendar-date pura e splittarla (`const [y,m,d] = value.split("-")`).

3. **major** — `src/components/analytics/payment-breakdown.tsx` + `revenue-sparkline.tsx` — Mancano file di test. CLAUDE.md regola 7: "Every new file with logic must have a corresponding test file". I componenti hanno logica testabile: mapping label (`METHOD_LABELS[e.method] ?? e.method`), trasformazione `revenueCents / 100`, empty state. Decidere: (a) aggiungere `payment-breakdown.test.tsx` + `revenue-sparkline.test.tsx` con render minimi via Testing Library; (b) escludere recharts wrapper in `sonar.coverage.exclusions` + `vitest.config.ts` con commento "rendering SVG ricaths non testabile in jsdom" e estrarre le pure functions di mapping in un file separato testato.

4. **major** — `src/components/analytics/revenue-sparkline.tsx` (tutto il file) — Nessuno stato empty: se `data.length === 0` (Batch 2 finding #2 — silent error masking che propaga array vuoto, oppure davvero zero scontrini nel range) il chart renderizza un quadrato vuoto senza placeholder. `payment-breakdown.tsx:32-38` ha già il fallback "Nessuno scontrino nel periodo selezionato." → portarlo identico in sparkline per coerenza UX.

Se le action falliscono (vedi Batch 2 finding #2), l'utente vede: spinner → spinner sparisce → zeri/grafici vuoti. ### Batch 4 — Plans gating + Pro helpers

1. **major** — `src/server/profile-actions.ts:118-162` — Ordine `validation → ownership → rate-limit → UPDATE`. Per `updateBusiness` la validazione (string length, regex) precede l'ownership check (DB query). Un attaccante con un auth token valido può spammare ownership check (un round-trip DB ciascuno) senza incappare nel rate limit, che arriva dopo. CLAUDE.md regola 29 (double-gate prima di expensive ops) suggerisce: rate limit → ownership → validation → UPDATE. 2. **major** — `src/server/profile-actions.ts:222-259` — `changePassword`: `signInWithPassword` (riga 225) può ruotare il session cookie di Supabase. Conseguenze possibili: (a) l'utente viene loggato fuori dal device corrente (UX rotta), o (b) la session originale resta valida insieme alla nuova → un attaccante con session pre-cambio resta autenticato. CLAUDE.md non copre lo scenario. Verificare con Supabase docs (session refresh semantics) e aggiungere un test E2E che (1) emette login, (2) cambia password, (3) verifica che la session pre-cambio sia invalidata e quella post-cambio sia valida.
